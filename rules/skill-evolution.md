# Skill evolution — learn daily, stay lean

The skill and rules are living documents. Improve them continuously from real
usage — without letting them bloat.

## Extract new steering

- At the end of each working day/session, scan the dialogue for corrections
  the human had to make (wrong assumption challenged, missed coverage, quality
  bar raised, process redirected). Each correction the skill failed to
  pre-empt is a candidate rule.
- Write candidates **abstract**: the principle once, concrete details demoted
  to a parenthetical example — one rule that covers all stacks and tools, not
  a stack-specific copy.
- A correction must recur or be clearly generalizable before it earns a rule;
  one-off situational feedback stays in the project's notes, not the skill.

## Anti-bloat discipline (applies to every edit)

- **Merge, don't append**: before adding, find the existing rule that almost
  covers it and sharpen that rule instead. One file/section per cohesive
  topic; no duplicates — a new realization updates, it doesn't multiply.
- **Prune stale rules**: models improve. Periodically re-evaluate each rule
  against current model behavior; when frontier and mid-tier models reliably
  do something unprompted (built-in intelligence), delete the rule — it is
  spent tokens and noise. Version history preserves it.
- **Token-optimize after edits**: re-read the touched file and compress
  without losing information — every session pays for these tokens. Concise
  beats verbose; a sentence beats a paragraph.
- **Net-size budget**: an edit that grows the skill must justify each added
  line with a failure it prevents. Prefer edits that hold size constant or
  shrink it. Never pad, restate, or add filler.

## Validate before keeping

A changed rule should demonstrably change behavior: spot-check on a real task
(A/B where cheap) that the model now does unprompted what the rule demands.
A rule that doesn't move behavior gets rewritten or removed.
